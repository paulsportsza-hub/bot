# MzansiEdge — Canonical Design Gallery

**Purpose**: this directory is the design source-of-truth for every card image
the bot renders. Paul and Claude iterate on each entry until it is "canonical"
(visually approved). Downstream implementer agents are then tasked to make the
live bot's `card_templates/*.html` reproduce these images cell-for-cell.

## Two-gallery model

| Gallery | URL | Generated by | Source of truth | Lifecycle |
|---|---|---|---|---|
| `latest/` | `https://mzansiedge.co.za/qa-gallery/latest/` | `bot/scripts/generate_qa_gallery.py` (auto, scans `card_templates/*.html`) | **Live behaviour** — what the bot produces today | Regenerated on demand (overwrites `index.html`) |
| `canonical/` | `https://mzansiedge.co.za/qa-gallery/canonical/` | Hand-curated by Paul + Claude | **Design intent** — what the bot *should* produce | Survives every regen; only changes when Paul approves |

## Workflow

1. **Iterate**: Paul and Claude work on a card draft (HTML in `/tmp/`, sample
   data in evidence dir, render via existing Playwright pipeline). Once a state
   is approved, the PNG lands in `canonical/<template_name>_<variant>.png`.
2. **Capture rationale**: each entry in `index.html` carries a "design notes"
   block with the locked constraints (brand tokens, layout rules, banned
   variants). This is the spec the implementer agent consumes.
3. **Cross-reference**: the canonical entry links back to the brief / INV /
   FIX where it was approved, and forward to the file:line targets in
   `card_templates/` and `card_data.py` where the live implementation lives.
4. **Implement**: a downstream agent (or a FIX brief) is tasked to make the
   live `card_templates/<template_name>.html` produce a render that matches
   the canonical PNG. Acceptance test: `latest/<template_name>.png` is
   visually identical to `canonical/<template_name>.png` (within OCR tolerance).
5. **Diff loop**: if the live render drifts from canonical, regenerate
   `latest/`, eyeball-diff against `canonical/`, file a FIX.

## Status badges (used in `index.html`)

- `✅ Canonical` — Paul has approved this design; live bot must match.
- `🔄 In review` — design draft pending Paul's approval; do not implement yet.
- `📝 Draft` — Claude exploring options; not for review yet.
- `⚠ Needs rework` — was canonical, but a constraint changed; design must be
  re-iterated before the implementer agent picks it up.

## Constraints downstream agents must honour

- **Do not** edit canonical PNGs to match the live bot — it is the other way
  around. If the live bot diverges, fix the live bot, never the canonical.
- **Do not** delete a canonical entry without Paul's explicit approval, even
  if the corresponding template no longer exists. Mark `⚠ Needs rework`
  instead, with a note explaining why.
- **Do not** rename the PNG files unless Paul approves — file naming carries
  semantic meaning that other agents grep for.

## Naming convention

- Single-state cards: `<template_name>.png` (e.g. `edge_detail.png`)
- Multi-state cards: `<template_name>_<variant>.png`
  (e.g. `edge_picks_index_bronze.png`, `edge_picks_index_silver.png`, ...)
- File extension: always `.png`.
- Variant tokens are lowercase, hyphenless (use underscores).

## Where the canonical gallery does NOT belong

- It is **not a regression baseline** for unit tests — that is what
  `tests/snapshots/*.png` is for (per-render byte-exact). Canonical is
  visual + curatorial; snapshots are mechanical.
- It is **not a release artefact** — it is a design document.
